# Tests

The tests included here all assume the same testing harness and apparatus. They
target actual machines as much as possible and have varying degrees of
complexity.

These tests assert value equality for 8-bit and 16-bit register values. They do
this via two interrupts. `0x22` will assert that `AL` is equal to `AH` and
`0x23` will assert that `AX` is equal to `BX`.

It is up to the emulator or hardware author to create an apparatus to handle
the `INT` instruction calls to actually perform the comparison and trigger
whatever signaling mechanism would be appropriate. For an emulator, for
instance, one might just 'emulate' the `INT` instruction to simply compare
the registers when it sees the vector requested as those two vector indices.
To test on actual hardware, one might install, before running the tests,
interrupt handlers into those two vectors that will signal appropriately.

The tests themselves do set up their own stack and assume they will be loaded
in as `0x100`, which is the typical entrypoint address for a DOS `.COM` file.

## Simple Tests

These tests are for some of the more basic instructions. These tests generally
have no branches and try to use as few other kinds of instructions as possible.

## Complex Tests

These tests are more comprehensive tests for each instruction. They use a wide
variety of instructions and behavior to test instruction behavior and flags.

## AI Usage

Generating test cases is very human-error prone and involves a lot of copying
and pasting. You'll generally write very basic tests that assert correct
behavior and forget that you should also test invalidating behavior. As such,
AI was used to create most of the comprehensive test cases.

Most of the `complex` tests were generated by an AI. In this case, I explored
just using the ChatGPT 5 "Thinking" mode to see how it would do. It generally
produced the diverse use-cases you would want but with a ton of mistakes. I
usually went through and fixed all of the broken tests.

The AI used (GPT5 circa September 2025) would fail at a variety of things related to
correct assembly code.

1. It would forget that it cannot use ALU instructions between the operation it was testing and the assertions for flags.
2. It would forget that certain macros it wrote would clobber registers it wants to test later
3. It would forget to include labels for testing backward jumps not being taken, as though it decided to include such a backward jump to invalid the assertion but, by the nature of an LLM, it cannot go back and create the label and the relevant code.
4. It would just put data in between tests which you would then arbitrarily run.
5. It would absolutely just forget how the CPU flags work and what their expected result should be. Generally, the parity flag (PF) would really confuse it since that requires counting the number of `1`'s in the least-significant byte of the result, and the AI cannot count to save its life.

In no circumstance did the AI produce a perfect test file.
